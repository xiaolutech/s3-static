# ADR-2025-08-31: 可配置缓存策略系统实现完成

## 状态
已完成 - 2025年8月31日

## 功能概述

成功实现了一个完整的可配置 HTTP 缓存策略系统，替代了原有的固定缓存策略。该系统支持三种缓存策略：`no-cache`、`max-age` 和 `immutable`，并将默认策略设置为更安全的 `no-cache`。此功能已在整个代码库中完整实现，包括配置管理、HTTP 处理器、测试覆盖和文档支持。

## 技术选择

### 核心技术栈
- **Go 1.25.0**: 利用现代 Go 特性实现类型安全的配置管理
- **环境变量配置**: 通过 `CACHE_STRATEGY` 环境变量进行运行时配置
- **HTTP 标准合规**: 严格遵循 HTTP/1.1 缓存规范 (RFC 7234)
- **S3 兼容性**: 保持与 AWS S3 缓存行为的完全兼容性

### 配置验证机制
- **编译时类型安全**: 使用 Go 的类型系统确保配置正确性
- **运行时验证**: 内置配置验证确保只接受有效的缓存策略值
- **默认值策略**: 采用安全优先的默认配置

## 架构设计

### 1. 配置层 (internal/config)
```go
type Config struct {
    // 缓存配置
    DefaultCacheDuration time.Duration `env:"CACHE_DURATION"`
    CacheStrategy        string        `env:"CACHE_STRATEGY"` // "no-cache", "max-age", or "immutable"
}

// 验证逻辑
validCacheStrategies := map[string]bool{
    "no-cache":  true,
    "max-age":   true,
    "immutable": true,
}
```

### 2. HTTP 处理层 (internal/handler)
```go
func (h *FileHandler) setCacheControlHeader(w http.ResponseWriter, path string) {
    switch h.config.CacheStrategy {
    case "no-cache":
        w.Header().Set("Cache-Control", "no-cache")
    case "max-age":
        w.Header().Set("Cache-Control", fmt.Sprintf("max-age=%d", int(h.config.DefaultCacheDuration.Seconds())))
    case "immutable":
        w.Header().Set("Cache-Control", fmt.Sprintf("max-age=%d, immutable", int(h.config.DefaultCacheDuration.Seconds())))
    }
}
```

## 设计原理

### 1. 安全优先原则
- **默认 no-cache**: 选择最安全的缓存策略作为默认值，确保数据一致性
- **显式配置**: 需要更激进缓存策略时必须显式配置
- **向后兼容**: 现有部署可通过环境变量保持原有行为

### 2. 配置驱动架构
- **环境变量驱动**: 支持容器化部署和不同环境配置
- **运行时可配置**: 无需重新编译即可调整缓存策略
- **配置验证**: 启动时验证配置有效性，快速失败原则

### 3. HTTP 标准合规性
- **RFC 7234 兼容**: 严格遵循 HTTP 缓存规范
- **条件请求支持**: 完整支持 ETag 和 Last-Modified 条件请求
- **304 Not Modified**: 优化网络传输效率

### 4. 缓存策略语义
- **no-cache**: 强制验证缓存，适用于可变内容
- **max-age**: 基于时间的缓存，适用于相对稳定的内容
- **immutable**: 不可变内容缓存，适用于版本化资源

## 替代方案

### 方案 1: 硬编码缓存策略
**优点**: 
- 实现简单，性能开销最小
- 行为可预测

**缺点**: 
- 缺乏灵活性，无法适应不同场景
- 修改策略需要重新编译和部署

**决策**: 被拒绝，因为不满足多环境部署需求

### 方案 2: 基于文件类型的自动策略选择
**优点**: 
- 智能化策略选择
- 减少配置复杂性

**缺点**: 
- 增加系统复杂性和维护成本
- 策略选择逻辑可能不符合实际需求
- 调试困难

**决策**: 被推迟，可作为未来增强功能

### 方案 3: 基于路径模式的细粒度配置
**优点**: 
- 最大化的配置灵活性
- 支持复杂的缓存需求

**缺点**: 
- 配置复杂度过高
- 增加运维负担
- 当前需求不需要如此复杂的配置

**决策**: 被推迟，当前简单配置已满足需求

## 影响和权衡

### 正面影响
1. **数据一致性提升**: 默认 `no-cache` 策略确保客户端获取最新数据
2. **部署灵活性**: 支持不同环境使用不同缓存策略
3. **标准合规性**: 更好地遵循 HTTP 缓存最佳实践
4. **性能优化**: 通过条件请求减少不必要的数据传输
5. **运维友好**: 缓存问题更容易诊断和解决

### 权衡考虑
1. **性能影响**: `no-cache` 策略可能增加服务器负载（通过条件请求缓解）
2. **网络开销**: 更频繁的条件请求验证（但避免了完整内容传输）
3. **配置复杂性**: 需要理解不同缓存策略的适用场景

### 迁移影响
1. **向后兼容**: 现有部署需要显式设置 `CACHE_STRATEGY=max-age` 保持原有行为
2. **测试更新**: 所有相关测试已更新以支持新的缓存策略
3. **文档完善**: 提供了详细的缓存策略配置文档

## 实现细节

### 配置管理
- **默认配置**: `CacheStrategy: "no-cache"`
- **环境变量**: `CACHE_STRATEGY` 支持运行时配置
- **验证机制**: 启动时验证策略有效性

### HTTP 处理器集成
- **策略应用**: 在 `setS3Headers` 方法中根据配置设置 Cache-Control 头
- **条件请求**: 保持现有的 ETag 和 Last-Modified 处理逻辑
- **错误处理**: 无效策略使用默认的 `no-cache` 策略

### 测试覆盖
- **配置测试**: 验证配置加载和验证逻辑
- **处理器测试**: 验证缓存头设置正确性
- **集成测试**: 端到端缓存行为验证
- **工厂测试**: 确保配置正确传递到存储层

## 部署配置示例

### Docker Compose 配置
```yaml
services:
  s3-static:
    image: s3-static:latest
    environment:
      - CACHE_STRATEGY=no-cache        # 推荐：可变内容
      # - CACHE_STRATEGY=immutable     # 可选：不可变内容
      # - CACHE_DURATION=8760h         # 配合 immutable 使用
    ports:
      - "8080:8080"
```

### 环境变量配置
```bash
# 推荐配置（可变内容）
export CACHE_STRATEGY=no-cache

# 不可变内容配置
export CACHE_STRATEGY=immutable
export CACHE_DURATION=8760h  # 1年

# 传统配置（不推荐）
export CACHE_STRATEGY=max-age
export CACHE_DURATION=1h
```
