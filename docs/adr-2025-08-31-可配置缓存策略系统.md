# ADR-2025-08-31: 可配置缓存策略系统

## 状态
已接受 - 2025年8月31日

## 功能概述

实现了一个可配置的 HTTP 缓存策略系统，替代了原有的固定 `max-age` 缓存策略。该系统支持三种缓存策略：`no-cache`、`max-age` 和 `immutable`，并将默认策略从 `max-age` 改为更安全的 `no-cache`。

## 技术选择

### 核心技术栈
- **Go 1.25.0**: 利用现代 Go 特性实现配置管理
- **环境变量配置**: 通过 `CACHE_STRATEGY` 环境变量进行运行时配置
- **HTTP 标准**: 严格遵循 HTTP/1.1 缓存规范 (RFC 7234)
- **S3 兼容性**: 保持与 AWS S3 缓存行为的兼容性

### 配置系统
- **结构化配置**: 在 `Config` 结构体中添加 `CacheStrategy` 字段
- **环境变量驱动**: 支持通过 `CACHE_STRATEGY` 环境变量动态配置
- **验证机制**: 内置配置验证确保只接受有效的缓存策略值

## 架构设计

### 配置层 (internal/config)
```go
type Config struct {
    // 其他配置字段...
    DefaultCacheDuration time.Duration `env:"CACHE_DURATION"`
    CacheStrategy        string        `env:"CACHE_STRATEGY"` // "no-cache", "max-age", or "immutable"
}
```

### 处理层 (internal/handler)
- **策略应用**: 在 `setS3Headers` 方法中根据配置应用相应的缓存策略
- **条件请求**: 保持现有的 ETag 和 Last-Modified 条件请求处理逻辑
- **S3 兼容性**: 确保缓存头设置符合 S3 API 规范

### 支持的缓存策略
1. **no-cache**: 强制浏览器每次都验证缓存（默认策略）
2. **max-age**: 基于时间的缓存策略，使用 `DefaultCacheDuration` 配置
3. **immutable**: 不可变内容缓存，适用于版本化资源

## 设计原理

### 1. 安全优先原则
- **默认 no-cache**: 选择最安全的缓存策略作为默认值
- **显式配置**: 需要更激进缓存策略时必须显式配置
- **数据一致性**: 确保客户端始终获取最新数据

### 2. 灵活性与可配置性
- **运行时配置**: 无需重新编译即可调整缓存策略
- **环境适应**: 不同部署环境可使用不同的缓存策略
- **渐进式优化**: 可根据实际使用情况调整缓存策略

### 3. HTTP 标准合规性
- **RFC 7234 兼容**: 严格遵循 HTTP 缓存规范
- **条件请求支持**: 保持 ETag 和 Last-Modified 的完整支持
- **304 Not Modified**: 优化网络传输效率

### 4. S3 API 兼容性
- **AWS S3 行为**: 模拟 AWS S3 的缓存头行为
- **标准响应头**: 保持 `x-amz-request-id` 等 S3 标准头
- **错误处理**: S3 兼容的错误响应格式

## 替代方案

### 方案 1: 保持固定 max-age 策略
**优点**: 
- 实现简单，无需额外配置
- 更好的性能表现

**缺点**: 
- 缺乏灵活性
- 可能导致数据不一致问题
- 不适合动态内容场景

**决策**: 被拒绝，因为缺乏灵活性且存在数据一致性风险

### 方案 2: 基于文件类型的自动策略选择
**优点**: 
- 智能化的缓存策略选择
- 无需手动配置

**缺点**: 
- 增加系统复杂性
- 难以预测的行为
- 调试困难

**决策**: 被推迟，可作为未来增强功能

### 方案 3: 基于路径的缓存策略配置
**优点**: 
- 细粒度控制
- 适合复杂场景

**缺点**: 
- 配置复杂
- 增加维护成本
- 过度工程化

**决策**: 被推迟，当前需求不需要如此复杂的配置

## 影响和权衡

### 正面影响
1. **数据一致性提升**: 默认 `no-cache` 策略确保客户端获取最新数据
2. **配置灵活性**: 支持不同部署环境的缓存需求
3. **标准合规性**: 更好地遵循 HTTP 缓存标准
4. **调试友好**: 缓存问题更容易诊断和解决

### 权衡考虑
1. **性能影响**: `no-cache` 策略可能增加服务器负载
2. **网络开销**: 更频繁的条件请求验证
3. **配置复杂性**: 需要理解不同缓存策略的适用场景

### 迁移影响
1. **向后兼容**: 现有部署需要显式设置 `CACHE_STRATEGY=max-age` 保持原有行为
2. **测试更新**: 需要更新集成测试以适应新的默认行为
3. **文档更新**: 需要更新部署和配置文档

## 实现细节

### 配置验证
```go
validCacheStrategies := map[string]bool{
    "no-cache":  true,
    "max-age":   true,
    "immutable": true,
}
```

### 缓存头设置逻辑
```go
// 当前实现 (临时硬编码)
w.Header().Set("Cache-Control", "no-cache")

// 计划实现 (基于配置)
switch h.config.CacheStrategy {
case "no-cache":
    w.Header().Set("Cache-Control", "no-cache")
case "max-age":
    w.Header().Set("Cache-Control", fmt.Sprintf("max-age=%d", int(h.config.DefaultCacheDuration.Seconds())))
case "immutable":
    w.Header().Set("Cache-Control", fmt.Sprintf("max-age=%d, immutable", int(h.config.DefaultCacheDuration.Seconds())))
}
```
